## Step 1 — 구독 매니저 (PR-a, PR-b)

트리거: 사용자 질문 *"브라우저 탭 3~4개에 LENS 페이지 각각 켜두면 어떻게 돼?"*
→ ref-count 없는 한계가 정확히 노출되는 시나리오. 도서관 비유로 Tier 시스템
설명 → 사용자가 진행 결정.

**Tier 설계** (트레이딩 데스크 표준 패턴):
- T1 영구: 자체 포지션 leg, 워치리스트
- T2 ref-count + warm-down: 페이지가 보는 코드
- T3 1회 fetch: 일회성 lookup

**PR-a / PR-b 분리 이유**: PR-a (코드 단위 ref-count + warm-down)로
멀티탭 충돌 90% 해결. PR-b (client별 추적 + WS disconnect 자동 cleanup)는
강건성 보강. 점진적 릴리스로 회귀 위험 분산.

**warm-down 60초 결정**: 30초 vs 60초 에이전트 양립 토론.
- 30초 옹호: 짧은 왕복 5~20초 충분 커버, 슬롯 회수 2배 빠름, 디버깅 명료
- 60초 옹호: 트레이더 분석 패턴 30~50초 (차트 보고 메모하기), t1102 REST 재호출이 진짜 비용, 단일 사용자라 capacity 압박 없음
- 채택: **60초**, 단 `REALTIME_WARM_DOWN_SECS` env로 운영 중 조정 가능. 단일 사용자 환경 + t1102 burst 회피가 결정타.

**WS client_id 교환**: 연결 직후 `{"type":"hello","client_id":N}` 메시지 →
frontend가 store에 저장 → `usePageStockSubscriptions`가 `X-LENS-Client-Id`
헤더로 첨부 → realtime이 client_subs에 추적 → WS disconnect 시 그 client의
sub 자동 cleanup.

**WS 재연결 race fix**: clientId가 useEffect 의존성에 안 들어가면 재연결 시
구 client_subs는 disconnect cleanup으로 정리되고 새 client_subs는 비어있는
상태가 됨 → 데이터 흐름 끊김. 해결: clientId 변경 시 lastSet reset → 모든
코드 새 id로 재구독 (warm_resumed 경로로 LS 연결 유지).

## Step 2 — 활성 포지션 leg 영구 sub (Tier 1)

backend가 active 포지션 leg를 realtime의 `/permanent-stocks` set으로 push.
세 시점에 발사: startup, 포지션 등록/청산/삭제 후 (`asyncio.create_task`로
fire-and-forget). realtime은 diff 계산해서 SubscribeStocks/UnsubscribeStocks
발사 — PR-a/PR-b의 ref-count + warm-down 흐름 위에 자연스럽게 얹힘.

**Lost-update 방지**: 처음 read lock → diff → write lock 분리한 코드를
write lock 한 번에 통합 (검증 에이전트가 잡아냄).

**realtime 단독 재시작 시 공백** 허용: backend polling 자동 회복은 후속 PR.
다음 포지션 변경 시점까지 영구 sub 비어있는 짧은 gap.
